# Skill 设计清单与模板

## 设计清单

在封装 Skill 之前，确认以下 7 个问题都有答案：

- [ ] **触发场景**：这个 Skill 在什么情况下使用？
- [ ] **输入字段**：用户必须提供哪些信息？
- [ ] **工作流步骤**：AI 需要按什么顺序做什么？
- [ ] **输出格式**：输出必须包含哪些章节或字段？
- [ ] **示例**：给一个输入和预期输出样例
- [ ] **注意事项**：常见错误、边界条件、验证建议
- [ ] **测试**：自己先用一个真实任务跑一遍

---

## Anthropic SKILL.md 标准模板

```markdown
---
name: skill-name
description: 一句话说明何时使用这个 Skill
---

# Skill 名称

## 何时使用
描述使用场景和触发条件。

## 输入
描述需要用户提供什么信息。

## 工作流
分步骤说明 Skill 的执行流程。

## 输出格式
明确输出的结构、字段、格式要求。

## 示例
给一个输入 / 输出样例。

## 注意事项
列出常见错误、边界条件、验证建议。
```

## Hermes Skill 机制补充

Hermes 在 Anthropic 规范基础上扩展了以下能力：

| 能力 | 说明 |
|------|------|
| 定时任务 | Skill 可以被 cron 表达式触发 |
| 消息渠道 | Skill 可以通过飞书/钉钉/微信触发和推送 |
| MCP 工具 | Skill 可以调用浏览器、搜索、文件系统等工具 |
| 跨会话记忆 | Skill 执行结果可以被长期记忆，供后续调用 |

## 示例：硬件产品需求分析 Skill

```markdown
---
name: hardware-pm-requirement-analysis
description: 用于硬件产品 PRD 前的多 Agent 需求分析，通过用户、技术、商业、竞品四个视角交叉校验，降低 AI 幻觉。
---

# 硬件产品需求分析 Skill

## 何时使用
- 写 PRD 之前，需要对需求做系统性分析
- 团队对一个功能是否该做存在争议
- 需要多个视角（用户/技术/商业/竞品）交叉验证

## 输入
1. 产品名称与一句话描述
2. 目标用户画像
3. 产品目标与约束（预算、时间、技术边界）
4. 需要分析的功能或问题

## 工作流
1. 并行生成 4 个 Agent 的分析
2. 让 4 个 Agent 互相交叉校验
3. 由人整合输出最终需求分析结论

## 输出格式
Markdown 报告，包含：产品目标、多视角分析、交叉校验、争议点、优先级建议、不确定项。

## 示例
输入：我们要做一款带氛围灯功能的植物生长照射仪...
输出：...

## 注意事项
- 每个 Agent 必须引用证据或明确标注"推测"
- 最终结论必须由人做判断
- 对不确定项要给出具体的验证方法
```
