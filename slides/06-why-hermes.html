为什么用 Hermes 做演示

不是推荐你换工具，而是它的架构刚好能把今天讲的 4 条原则串起来。

演示选择 Hermes，是因为它的设计理念（长期记忆、Skill 封装、定时任务、消息渠道）和今天的方法论高度吻合。Claude Code、Codex 也是非常强的 AI 工具，只是更偏向与代码仓库深度协作。以下对比只从「产品工作自动化」视角出发，方便理解，不构成选型建议。
维度 Hermes Claude Code OpenAI Codex ChatGPT / Claude Web
设计目标 长期在线的自主 Agent 终端编码助手 终端编码 Agent 单次对话工具
持久记忆 跨会话自动保留 项目内上下文记忆 云端会话内上下文，重启后需重建 每段对话独立
封装复用 Skill 系统，PM 也能写 Skills / Hooks Skills / 云 Agent 无原生封装机制
定时任务 自托管 cron，离线运行 云托管 / 会话级循环 云托管任务 无定时能力
消息渠道 10+ 平台原生接入 有限预览 无 仅官方 App/网页
最适合 PM 的环节 调研、分析、自动化闭环 代码重构、代码审查 代码生成、长时开发任务 头脑风暴、文案
基于 2026 年 7 月公开信息整理。AI 工具能力迭代快，各产品实际能力请以官方最新文档为准。表格只反映从「产品工作自动化」视角的观察。

01 记忆

项目背景不需要每次重新交代，AI 越用越懂你。

02 封装

把调研、分析流程写成 Skill，下次直接调用。

03 定时

Reddit 情报、周报这类重复工作可以自动跑。

04 推送

结果自动发到飞书/钉钉/Slack，你只做决策。
